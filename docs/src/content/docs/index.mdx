## Why gokeel

<CardGrid stagger>
	<Card title="Declarative transactions" icon="seti:go">
		A context-bound unit of work over `database/sql`, modeled on Spring's
		`@Transactional`: `Run` commits on success and rolls back on error or panic.
	</Card>
	<Card title="Propagation & savepoints" icon="random">
		Nested calls join the active transaction or open a savepoint according to
		the configured propagation, just like a Spring service graph.
	</Card>
	<Card title="Commit synchronizations" icon="approve-check">
		Register before-commit, after-commit, and after-completion callbacks that
		run in the same phases Spring exposes.
	</Card>
	<Card title="In-process event bus" icon="list-format">
		A small, synchronous event bus delivers each event to the listeners
		registered for its type, in registration order, with panic isolation.
	</Card>
	<Card title="Transactional outbox" icon="seti:db">
		Events are written in the producing transaction and delivered after commit,
		so a publication is never lost or duplicated relative to the state change.
	</Card>
	<Card title="Hierarchical log levels" icon="setting">
		Spring Boot's `logging.level` model for `log/slog`: one assignment on a
		package path governs every logger beneath it, adjustable at runtime.
	</Card>
	<Card title="Zero-dependency cores" icon="seti:lock">
		The `transaction`, `eventbus`, and `logging` modules use only the Go
		standard library, a property enforced in CI.
	</Card>
</CardGrid>
